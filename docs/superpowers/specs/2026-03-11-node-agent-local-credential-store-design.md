# Node Agent Local Credential Store Design

**Date:** 2026-03-11

**Objective:** Add a first-party Node-only local credential store for Evory Agents so CLI and script callers can discover, persist, promote, replace, and clear the canonical local Agent credential without mixing browser and filesystem responsibilities.

## Scope

This change covers:

- adding a Node-only local credential store module
- defining one canonical persisted file shape for the local Agent record
- implementing deterministic discovery precedence across env, canonical file, and compatibility fallback sources
- supporting local state transitions for `pending_binding` and `bound`
- supporting canonical file replacement after key rotation
- adding focused tests for discovery, writes, transitions, and failure handling

This change does not cover:

- front-end control plane writes to local files
- browser access to `~/.config`
- remote validation requests such as `GET /api/agent/tasks`
- multi-profile support
- a full interactive CLI

## Problem Statement

Evory now has a clearer public contract for canonical local credential storage, but the repository still lacks a first-party implementation that Node-based callers can reuse. Today, each caller would need to reimplement:

- `EVORY_AGENT_API_KEY` override handling
- canonical file path resolution for `~/.config/evory/agents/default.json`
- fallback reads from `.env.local` and `.evory/agent.json`
- write restrictions so only the canonical file is mutated
- local lifecycle transitions from `pending_binding` to `bound`

Leaving those decisions to each caller would recreate the same ambiguity the public docs just removed.

## Approaches Considered

### 1. Minimal helper functions

Provide only path helpers plus raw read and write primitives.

Pros:

- smallest implementation
- low initial surface area

Cons:

- pushes precedence, warnings, and write restrictions into every caller
- likely duplicates logic across future scripts

### 2. Recommended: a small store module with high-level operations

Provide one Node-only module that owns discovery precedence, local record parsing, write restrictions, and local lifecycle transitions.

Pros:

- one source of truth for local credential semantics
- easier to test and reuse
- supports future CLI work without committing to it now

Cons:

- slightly larger initial API surface

### 3. Full CLI subsystem first

Bundle storage, migration, printing, and operator workflows into one command-line tool.

Pros:

- best operator experience
- easiest to demo end-to-end

Cons:

- too much scope for the current need
- mixes storage semantics with CLI UX decisions

## Recommended Approach

Use approach 2.

Add a single Node-only module that owns:

- discovery precedence
- canonical record parsing
- warning classification for compatibility fallback sources
- writes to the canonical file
- local lifecycle transitions

The module should stay intentionally ignorant of remote Agent validation. Its job is local state management only.

## Architecture

### Module Boundary

Create a focused Node-only module at:

- `src/lib/agent-local-credential.ts`

It should import Node APIs such as:

- `node:fs/promises`
- `node:path`
- `node:os`

No React, browser, or route code should depend on it directly in this phase.

### Canonical Record Shape

The canonical persisted record is:

```json
{
  "agentId": "agt_xxx",
  "apiKey": "evory_xxx",
  "bindingStatus": "pending_binding",
  "updatedAt": "2026-03-11T00:00:00.000Z"
}
```

Field rules:

- `agentId` is required and represents the stable Evory Agent identity
- `apiKey` is required and represents the rotatable bearer secret
- `bindingStatus` is required and must be either `pending_binding` or `bound`
- `updatedAt` is required and must be an ISO timestamp generated by the module

### Discovery Precedence

The module should resolve a local credential in this order:

1. `EVORY_AGENT_API_KEY`
2. `~/.config/evory/agents/default.json`
3. `.env.local`
4. `.evory/agent.json`

The module should return both the discovered record and metadata describing the source.

Suggested source values:

- `env_override`
- `canonical_file`
- `dotenv_fallback`
- `project_file_fallback`
- `none`

### Write Restrictions

Only the canonical user-level file may be written or deleted by this module.

That means:

- `EVORY_AGENT_API_KEY` is read-only
- `.env.local` is read-only
- `.evory/agent.json` is read-only
- only `~/.config/evory/agents/default.json` is writable

If discovery resolves to a non-canonical source, the module should still return the record, but also surface:

- `writable: false`
- a warning that the source is compatibility-only and should be migrated

### Public API

The module should expose high-level functions only:

- `discoverAgentCredential(options?)`
- `savePendingAgentCredential(input)`
- `promoteAgentCredentialToBound(agentId)`
- `replaceAgentCredential(input)`
- `clearAgentCredential(agentId?)`

#### `discoverAgentCredential(options?)`

Returns a discriminated result that includes:

- `source`
- `writable`
- `record` when found
- `warnings`
- `error` when the canonical file exists but is structurally invalid

#### `savePendingAgentCredential(input)`

Writes a canonical record with:

- provided `agentId`
- provided `apiKey`
- `bindingStatus: "pending_binding"`
- generated `updatedAt`

#### `promoteAgentCredentialToBound(agentId)`

Loads the canonical file and transitions:

- `pending_binding` -> `bound`

It must reject if:

- the canonical file is missing
- the canonical record is invalid
- the stored `agentId` differs from the requested one

#### `replaceAgentCredential(input)`

Used after remote key rotation. It must:

- preserve the same `agentId`
- replace `apiKey`
- keep or accept the existing `bindingStatus`
- refresh `updatedAt`

It must reject if the stored canonical file belongs to a different Agent identity.

#### `clearAgentCredential(agentId?)`

Deletes only the canonical file. If `agentId` is supplied and does not match the stored record, it should reject rather than clear the wrong identity.

### Error Handling

#### Canonical file behavior

- missing canonical file => not an error; return `source: "none"` unless another source resolves
- invalid JSON or structurally invalid canonical file => structured error such as `invalid_canonical_file`
- canonical file errors are never silently swallowed

#### Compatibility fallback behavior

- if `.env.local` or `.evory/agent.json` cannot be parsed, discovery continues
- these failures surface as warnings rather than fatal errors
- discovery should keep scanning lower-priority sources only when the current fallback source is unusable

### Filesystem Semantics

Canonical writes should:

- ensure the parent directory exists
- write to a temporary file in the same directory
- rename into place atomically

This reduces the chance of partial writes leaving a corrupted canonical record.

## Testing Strategy

Add a dedicated test file:

- `src/lib/agent-local-credential.test.ts`

Test categories:

### Discovery precedence

- env override wins over every file source
- canonical file wins over compatibility fallback sources
- `.env.local` wins over `.evory/agent.json` when canonical is absent
- discovery returns `none` when nothing exists

### Write semantics

- `savePendingAgentCredential` writes the canonical file and parent directory
- writes generate `updatedAt`
- writes never touch fallback sources

### Lifecycle transitions

- `promoteAgentCredentialToBound` updates only `bindingStatus`
- `replaceAgentCredential` updates `apiKey` and `updatedAt` while preserving `agentId`
- `clearAgentCredential` removes only the canonical file

### Failure handling

- invalid canonical JSON returns a structured error
- invalid fallback content yields warnings and continues
- non-canonical sources return `writable: false`
- `promote` and `replace` reject identity mismatches

## Risks And Mitigations

- Local path handling may be OS-sensitive.
  - Mitigation: centralize all path construction and test with mocked homedir and cwd.
- Compatibility fallback parsing may become too clever.
  - Mitigation: keep fallback parsers minimal and purpose-built for this phase.
- Callers may assume discovery implies remote validity.
  - Mitigation: name APIs around local credential state only and document that remote validation is separate.

## Success Criteria

This work is successful when:

- Node callers can use one shared module instead of reimplementing local credential logic
- the module enforces the canonical write path and read-only compatibility sources
- the persisted record shape matches the published contract
- tests lock precedence, transitions, and failure modes
